# figma-cli 설계문서

## 1. 개요

Figma Desktop MCP 서버(로컬 127.0.0.1:3845)와 통신하는 Node.js CLI 도구.
Microsoft Playwright CLI 아키텍처를 벤치마킹하여 모듈화된 구조를 채택한다.

### 핵심 철학: CLI as a Skill

- MCP 스키마를 LLM이 직접 로딩하는 대신, CLI 명령어를 '스킬'로 활용
- 토큰 효율적: `figma extract <url>` 한 줄로 복잡한 MCP 워크플로우 완료
- 구조화된 JSON 출력으로 에이전트 파싱 용이

## 2. 아키텍처

### 2.1 계층 분리

```
┌─────────────────────────────┐
│  CLI Layer (commander.js)   │  인자 파싱, 검증, 도움말
├─────────────────────────────┤
│  Command Handlers           │  비즈니스 로직, 출력 포매팅
├─────────────────────────────┤
│  Service Layer              │  FigmaMCPClient (추상화)
│  (FigmaMCPClient)           │
├─────────────────────────────┤
│  Transport Layer            │  HTTP JSON-RPC, 세션 관리
│  (@modelcontextprotocol/sdk)│
└─────────────────────────────┘
        ↕ HTTP POST
┌─────────────────────────────┐
│  Figma Desktop MCP Server   │  127.0.0.1:3845/mcp
└─────────────────────────────┘
```

### 2.2 통신 프로토콜 (실측 검증 완료: 2026-02-11)

- **주 엔드포인트**: `POST http://127.0.0.1:3845/mcp` (JSON-RPC 2.0)
- **SSE 엔드포인트**: `http://127.0.0.1:3845/sse` (레거시, deprecated 예정 → 사용하지 않음)
- **세션 관리**: `mcp-session-id` 응답 헤더에서 획득, 이후 요청 헤더에 포함
- **세션 캐싱**: `~/.figma-cli/session.json`에 저장하여 반복 핸드셰이크 방지

#### 필수 요청 헤더

```
Content-Type: application/json
Accept: application/json, text/event-stream   ← 반드시 두 타입 모두 포함 (누락 시 406 에러)
mcp-session-id: <세션 ID>                     ← initialize 이후 모든 요청에 필수
```

#### 응답 형식 (Streamable HTTP)

`/mcp` 엔드포인트에 POST 요청해도 서버 응답의 Content-Type은 `text/event-stream`이다.
이는 MCP Streamable HTTP transport 특성으로, `/sse` 엔드포인트와는 무관하다.
파싱 시 `event: message\ndata: ` 접두사를 제거하고 JSON 파싱해야 한다.

```
HTTP/1.1 200 OK
Content-Type: text/event-stream     ← /mcp 응답이지만 SSE 형식
mcp-session-id: <세션 ID>

event: message
data: {"result": {...}, "jsonrpc": "2.0", "id": 1}
```

#### 연결 시퀀스

```
1. POST /mcp  → method: "initialize"           → 200 + mcp-session-id 헤더
2. POST /mcp  → method: "notifications/initialized"  → (알림, 응답 없음)
3. POST /mcp  → method: "tools/list"            → 사용 가능 도구 목록
4. POST /mcp  → method: "tools/call"            → 도구 실행
```

## 3. MCP 도구 상세 명세 (실측 검증 완료)

서버에서 제공하는 8개 도구의 실제 동작을 검증한 결과.

### 3.1 get_design_context

디자인 노드를 코드로 변환. 가장 핵심적인 도구.

- **응답 크기**: ~37KB (노드 복잡도에 비례, 타임아웃 주의)
- **반환**: 코드 문자열 (React 컴포넌트 + 에셋 URL)
- **에셋 URL**: `http://localhost:3845/assets/<hash>.svg` 형태로 로컬 서버에서 제공
- **content type**: `text`

| 파라미터 | 타입 | 필수 | 설명 |
|---------|------|------|------|
| nodeId | string | X | 노드 ID (미지정 시 현재 선택 노드) |
| clientLanguages | string | X | 프로그래밍 언어 (예: "typescript") |
| clientFrameworks | string | X | 프레임워크 (예: "react") |
| forceCode | boolean | X | 큰 노드에서도 코드 강제 반환 |
| artifactType | string | X | WEB_PAGE_OR_APP_SCREEN, COMPONENT_WITHIN_A_WEB_PAGE_OR_APP_SCREEN, REUSABLE_COMPONENT, DESIGN_SYSTEM |
| taskType | string | X | CREATE_ARTIFACT, CHANGE_ARTIFACT, DELETE_ARTIFACT |

### 3.2 get_metadata

노드 계층 구조를 XML로 반환. 경량 탐색용.

- **응답 크기**: ~1KB (get_design_context 대비 매우 작음)
- **반환**: XML 트리 (노드 ID, 이름, 타입, 위치/크기)
- **content type**: `text` (2개: XML 데이터 + get_design_context 호출 안내 메시지)

| 파라미터 | 타입 | 필수 | 설명 |
|---------|------|------|------|
| nodeId | string | X | 노드 ID (미지정 시 현재 선택 노드) |
| clientLanguages | string | X | 프로그래밍 언어 |
| clientFrameworks | string | X | 프레임워크 |

**응답 예시**:
```xml
<frame id="52:590" name="MCP 테스트 페이지" x="9879" y="960" width="360" height="989">
  <instance id="52:591" name="StatusBar" x="0" y="0" width="360" height="47" />
  <instance id="52:592" name="App_2depth" x="0" y="47" width="360" height="60" />
  ...
</frame>
```

### 3.3 get_screenshot

노드의 스크린샷을 캡처.

- **응답 크기**: ~38KB (해상도에 따라 변동)
- **반환**: Base64 인코딩 PNG 이미지
- **content type**: `image` (text가 아닌 image 타입)

| 파라미터 | 타입 | 필수 | 설명 |
|---------|------|------|------|
| nodeId | string | X | 노드 ID (미지정 시 현재 선택 노드) |
| clientLanguages | string | X | 프로그래밍 언어 |
| clientFrameworks | string | X | 프레임워크 |

### 3.4 get_variable_defs

디자인 토큰(변수) 정의를 반환. 색상, 폰트, 이펙트 등.

- **응답 크기**: ~2KB
- **반환**: JSON 문자열 (키: 토큰 이름, 값: 토큰 정의)
- **content type**: `text`

| 파라미터 | 타입 | 필수 | 설명 |
|---------|------|------|------|
| nodeId | string | X | 노드 ID (미지정 시 현재 선택 노드) |
| clientLanguages | string | X | 프로그래밍 언어 |
| clientFrameworks | string | X | 프레임워크 |

**응답 예시** (발췌):
```json
{
  "Text color/normal": "#0C1120",
  "Text color/accent": "#2D91FF",
  "Brand color/primary-regular": "#2D91FF",
  "Title/16px (T100-SemiBold)": "Font(family: \"Pretendard\", style: SemiBold, size: 16, weight: 600, lineHeight: 22, letterSpacing: 0)",
  "Shadow color/Floating Shadow": "Effect(type: DROP_SHADOW, color: #0000001A, offset: (0, 4), radius: 8, spread: 0)"
}
```

### 3.5 get_code_connect_map

Figma 노드와 코드 컴포넌트 간 매핑 조회.

- **응답 크기**: ~10B (매핑 없을 시 `{}`)
- **반환**: JSON (nodeId → {codeConnectSrc, codeConnectName})
- **content type**: `text`

| 파라미터 | 타입 | 필수 | 설명 |
|---------|------|------|------|
| nodeId | string | X | 노드 ID (미지정 시 현재 선택 노드) |
| clientLanguages | string | X | 프로그래밍 언어 |
| clientFrameworks | string | X | 프레임워크 |

### 3.6 add_code_connect_map

노드-코드 컴포넌트 매핑 등록. **쓰기 작업** (readOnlyHint: false).

| 파라미터 | 타입 | 필수 | 설명 |
|---------|------|------|------|
| nodeId | string | X | 노드 ID |
| source | string | **O** | 소스 코드 파일 경로 |
| componentName | string | **O** | 컴포넌트 이름 |
| label | string (enum) | **O** | 프레임워크 라벨 |
| clientLanguages | string | X | 프로그래밍 언어 |
| clientFrameworks | string | X | 프레임워크 |

**label 허용값**: React, Web Components, Vue, Svelte, Storybook, Javascript, Swift UIKit, Objective-C UIKit, SwiftUI, Compose, Java, Kotlin, Android XML Layout, Flutter, Markdown

### 3.7 create_design_system_rules

코드베이스 분석용 프롬프트를 반환. LLM이 디자인 시스템 규칙 문서를 생성하는 데 사용.

- **응답 크기**: ~2KB
- **반환**: 분석 가이드 프롬프트 텍스트 (토큰 정의, 컴포넌트 구조, 스타일링 방식 등 7개 항목)
- **content type**: `text`
- **특성**: 직접 데이터를 반환하는 것이 아니라, LLM이 코드베이스를 분석할 때 사용할 지시문을 제공

| 파라미터 | 타입 | 필수 | 설명 |
|---------|------|------|------|
| clientLanguages | string | X | 프로그래밍 언어 |
| clientFrameworks | string | X | 프레임워크 |

### 3.8 get_figjam

FigJam 파일의 노드를 코드로 변환. **FigJam 전용** (일반 Figma 파일에서 호출 시 에러).

- **에러 메시지**: "get_figjam is only available for FigJam files."
- **content type**: `text`
- **URL 형식**: `https://figma.com/board/:fileKey/:fileName?node-id=1-2`

| 파라미터 | 타입 | 필수 | 설명 |
|---------|------|------|------|
| nodeId | string | X | 노드 ID |
| clientLanguages | string | X | 프로그래밍 언어 |
| clientFrameworks | string | X | 프레임워크 |
| includeImagesOfNodes | boolean | X | 노드 이미지 포함 여부 (기본 true) |

## 4. 명령어 설계 (검증 결과 반영)

### 4.1 `figma extract [nodeId]` (핵심)
- **MCP 도구**: `get_design_context`
- **옵션**: `--format <react|html|vue>`, `--output <file>`, `--prompt <text>`, `--force-code`
- 디자인 노드를 코드로 변환, nodeId 미지정 시 현재 선택 노드

### 4.2 `figma inspect [nodeId]`
- **MCP 도구**: `get_metadata`
- **옵션**: `--json`
- 노드 계층 구조와 메타데이터를 XML/JSON으로 조회

### 4.3 `figma shot [nodeId]`
- **MCP 도구**: `get_screenshot`
- **옵션**: `--output <file>`
- 노드 스크린샷 캡처, Base64 → 파일 저장

### 4.4 `figma tokens [nodeId]`
- **MCP 도구**: `get_variable_defs`
- **옵션**: `--json`, `--filter <color|font|effect>`
- 디자인 토큰(색상, 폰트, 이펙트) 조회

### 4.5 `figma connect list [nodeId]`
- **MCP 도구**: `get_code_connect_map`
- 코드-디자인 매핑 정보 조회

### 4.6 `figma connect add`
- **MCP 도구**: `add_code_connect_map`
- **옵션**: `--source <path>`, `--name <componentName>`, `--label <framework>`
- 노드-코드 컴포넌트 매핑 등록 (쓰기 작업, 사용자 확인 필요)

### 4.7 `figma design-rules`
- **MCP 도구**: `create_design_system_rules`
- 디자인 시스템 규칙 문서 생성용 프롬프트 출력

### 4.8 `figma figjam [nodeId]`
- **MCP 도구**: `get_figjam`
- **옵션**: `--no-images`
- FigJam 파일 전용 코드 생성

## 5. FigmaMCPClient 설계

```typescript
class FigmaMCPClient {
  // 연결 관리
  connect(): Promise<void>
  disconnect(): Promise<void>
  isConnected(): boolean

  // 도구 래핑 메서드 (8개 도구 전체)
  extractDesign(nodeId?: string, options?: ExtractOptions): Promise<DesignContext>
  getMetadata(nodeId?: string): Promise<NodeMetadata>
  getScreenshot(nodeId?: string): Promise<Buffer>
  getVariableDefs(nodeId?: string): Promise<VariableDefs>
  getCodeConnectMap(nodeId?: string): Promise<CodeConnectMap>
  addCodeConnectMap(params: AddCodeConnectParams): Promise<void>
  createDesignSystemRules(): Promise<string>
  getFigJam(nodeId?: string, includeImages?: boolean): Promise<FigJamContext>

  // 세션 관리 (내부)
  private loadSession(): SessionInfo | null
  private saveSession(session: SessionInfo): void
  private reconnect(): Promise<void>

  // 응답 파싱 (내부) - SSE 형식 처리
  private parseSSEResponse(raw: string): JsonRpcResponse
}
```

**주의사항**:
- `nodeId`는 모든 도구에서 선택적 (미지정 시 Figma에서 현재 선택된 노드)
- `get_screenshot` 응답은 content type이 `image`이므로 별도 처리 필요
- `add_code_connect_map`은 유일한 쓰기 작업 → 사용자 확인 플로우 추가

## 6. 에이전트 데이터 전달 전략 (Playwright CLI 벤치마킹)

### 6.1 핵심 원칙

figma-dev-cli는 현재 명령어별로 다음 패턴을 사용한다:
- **텍스트/JSON 응답** → stdout 직접 출력 (`extract`, `inspect`, `tokens`, `connect`, `design-rules`, `figjam`)
- **이진 데이터 응답** → 파일 저장 후 경로 출력 (`shot`)

### 6.2 명령어별 전달 방식

| 명령어 | MCP 도구 | 응답 크기 | 전달 방식 | 이유 |
|--------|---------|----------|----------|------|
| `inspect` | get_metadata | ~1KB | **stdout** | 경량, 에이전트가 즉시 노드 ID 파악 |
| `tokens` | get_variable_defs | ~2KB | **stdout** | 경량, 토큰 값 즉시 참조 |
| `connect list` | get_code_connect_map | ~수십B | **stdout** | 경량 매핑 정보 |
| `design-rules` | create_design_system_rules | ~2KB | **stdout** | 프롬프트 텍스트, 즉시 사용 |
| `extract` | get_design_context | **~37KB+** | **stdout** | 코드/가이던스를 즉시 참조 |
| `shot` | get_screenshot | **~38KB+** | **파일(PNG) → 경로** | 이진 데이터, Base64 텍스트 전달 무의미 |
| `figjam` | get_figjam | 가변 | **stdout** | FigJam 코드 텍스트를 즉시 소비 |
| `connect add` | add_code_connect_map | ~수십B | **stdout** | 결과 메시지만 |

### 6.3 파일 저장 규칙

현재 구현에서 파일 저장은 `shot` 명령어에만 적용된다.
스크린샷은 `--output <path>`로 경로를 지정할 수 있고, 미지정 시 OS 임시 디렉토리에 저장된다.

```bash
# 코드/텍스트 응답은 stdout 직접 출력
figma-dev extract 52:590
figma-dev figjam 1:2

# 이진 응답(스크린샷)은 파일 저장
figma-dev shot 52:590 --output ./captures/52-590.png
# 미지정 시: /var/.../figma-52-590.png (OS 임시 디렉토리)
```

### 6.4 에이전트 워크플로우 예시

```
1. 에이전트: figma-dev inspect          → stdout에서 노드 트리 확인 (1KB)
2. 에이전트: figma-dev extract 52:590   → stdout에서 코드/가이던스 바로 확인
3. 에이전트: figma-dev tokens           → stdout에서 디자인 토큰 즉시 참조
4. 에이전트: figma-dev shot 52:590      → stdout에서 PNG 경로 확인 → Read로 이미지 확인
```

이 방식으로 에이전트는 매 단계마다 컨텍스트를 최소한으로 소모하면서, 필요한 데이터만 선택적으로 가져올 수 있다.

### 6.5 stdout 출력 포맷 (--json 모드)

모든 명령어는 `--json` 플래그를 지원한다. 기본은 사람 친화적 포맷.

**경량 데이터 (stdout 직접)**:
```json
{
  "status": "success",
  "data": { "XML 트리 또는 토큰 등 실제 데이터" },
  "metadata": { "nodeId": "52:590", "responseTime": "0.3s" }
}
```

**파일 출력 (`shot --json`)**:
```json
{
  "status": "success",
  "outputFile": "/tmp/figma-52-590.png",
  "outputSize": "38.1KB",
  "metadata": { "nodeId": "52:590" }
}
```

**에러**:
```json
{
  "status": "error",
  "error": {
    "code": "CONNECTION_FAILED",
    "message": "Figma Desktop MCP 서버에 연결할 수 없습니다. Figma Dev Mode가 활성화되어 있는지 확인하세요."
  }
}
```

## 8. 주요 설계 결정

1. **HTTP POST 우선**: SSE 대신 `/mcp` 엔드포인트를 주력으로 사용 (안정성)
2. **Accept 헤더 필수**: `application/json, text/event-stream` 두 타입 모두 포함 (누락 시 406)
3. **Streamable HTTP 응답 파싱**: `/mcp` 응답이 `text/event-stream` Content-Type으로 반환됨 (Streamable HTTP transport 특성, `/sse`와 무관). `event: message\ndata: ` 접두사 제거 후 JSON 파싱
4. **세션 캐싱**: `~/.figma-cli/session.json`에 세션 ID 저장하여 반복 핸드셰이크 방지
5. **nodeId 선택적**: 모든 도구에서 nodeId 미지정 시 Figma Desktop에서 현재 선택된 노드 사용
6. **fileKey 불필요**: 로컬 MCP 서버는 Desktop 앱 기준 동작, URL에서 fileKey 추출 불필요
7. **Playwright 패턴 차용**: commander.js + 모듈별 명령어 등록 함수 + 서비스 계층 분리
8. **zod 검증**: MCP 응답을 런타임에 스키마 검증하여 안정성 확보
9. **기본 타임아웃 60초**: get_design_context 응답이 클 수 있으므로 넉넉하게 설정. `--timeout` 또는 `FIGMA_CLI_TIMEOUT`으로 재정의
10. **에셋 URL 프록시**: get_design_context 응답에 포함된 `localhost:3845/assets/*` URL은 서버 실행 중에만 유효

```bash
curl -s -X POST http://127.0.0.1:3845/mcp \
  -H "Content-Type: application/json" \
  -H "Accept: application/json, text/event-stream" \
  -d '{"jsonrpc":"2.0","id":1,"method":"initialize","params":{...}}'
# → 200 OK
# → mcp-session-id: 86fb58c3-8ca9-40ce-88a1-94df0b4f2e66
```
